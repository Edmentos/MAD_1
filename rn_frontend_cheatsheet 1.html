React Native Frontend Cheatsheet

Only concepts taught in Dr. Downey's lab slides (Labs 1–9). Frontend only. Every section references which lab it came from.

1. What is React Native

Lab 1 — wk1_lab1

React = the core library. Handles component tree, state, virtual DOM diffing. Platform-agnostic.
React DOM = renderer for web browsers. Translates to HTML elements.
React Native = renderer for mobile. Maps components to native UI widgets (UIView on iOS, android.view.View on Android) via a bridge.

JSX code runs on a JavaScript virtual machine. During build, native code compiles to Java/C++ binaries. The JS is not compiled — it runs on the JS VM and communicates with native via JSON.

2. Core Components

Lab 3 — wk2_lab3

No HTML in React Native. Native devices don't have a DOM. You must use RN components, and every one must be explicitly imported from 'react-native'.
RN ComponentWeb EquivalentPurpose
<View><div>Container / layout box. The mobile equivalent of a div.
<Text><p> / <span>Display text. ALL text must be inside Text or the app crashes.
<Image><img>Display images. Local (require) or network (uri). Must set width/height for network images.
<TextInput><input>User text entry. Uses onChangeText and value props.
<Button><button>Platform-native button. Self-closing. Uses title prop not children. Looks like iOS/Android native button automatically.
<TouchableOpacity><button>Custom pressable. Fades on press. Can wrap anything.
<TouchableHighlight><button>Custom pressable. Shows underlay colour on press. Must have exactly one child.
<ScrollView>overflow: scrollScrollable container. Renders ALL children. Good for small content only.
<FlatList>—Efficient scrollable list. Only renders visible items. Use for long lists.
<ActivityIndicator>—Loading spinner.

Import pattern

import { View, Text, FlatList, TextInput, Button,
         TouchableOpacity, TouchableHighlight,
         StyleSheet, Alert, Image, ScrollView,
         ActivityIndicator } from 'react-native';

3. Styling — StyleSheet

Lab 3 — wk2_lab3

No CSS files in React Native. Styling is written in JavaScript using StyleSheet.create() or inline objects. Properties are camelCase. Values are numbers (no px). 
import { StyleSheet } from 'react-native';

const styles = StyleSheet.create({
  container: {
    flex: 1,
    padding: 16,
    backgroundColor: '#f5f5f5',
  },
  title: {
    fontSize: 20,
    fontWeight: 'bold',
    color: 'red',
  },
});

// Apply:
<View style={styles.container}>
<Text style={styles.title}>Hello</Text>

// Inline:
<View style={{ padding: 10, backgroundColor: 'blue' }}>

// Combine multiple (array syntax):
<Text style={[styles.title, { color: 'green' }]}>

// Conditional style:
<Text style={[styles.title, item.priority === 'High' && { color: 'red' }]}>
Why StyleSheet.create()? From slides: provides better auto-completion in VSCode and allows potential internal performance optimisations by React Native. Clearly separates JSX structure from styling logic.

4. Flexbox Layout

Lab 5 — wk3_lab5

Every <View> in RN is automatically a flex container. Default flexDirection is column (top to bottom). On web the default is row.
flexDirectionMain AxisCross Axis
column (default)Top → BottomLeft → Right
rowLeft → RightTop → Bottom
column-reverseBottom → TopRight → Left
row-reverseRight → LeftBottom → Top

justifyContent (main axis) and alignItems (cross axis)

// Values for both:
'center'  'flex-start'  'flex-end'
'space-between'  'space-around'  'space-evenly'
'stretch'  // alignItems default — children fill cross axis

flex (child sizing)

// flex: 1 → fill all remaining space
// flex: 2 vs flex: 1 → 2:1 ratio of free space
// No flex → only takes up as much space as content needs

<View style={{ flexDirection: 'row' }}>
  <View style={{ flex: 1, backgroundColor: 'red' }} />    // takes 1/3
  <View style={{ flex: 2, backgroundColor: 'blue' }} />   // takes 2/3
</View>

5. TextInput — Handling User Input

// onChangeText — fires every keystroke
// onSubmitEditing — fires when user presses enter/done
// value — binds state to input (two-way binding)

<TextInput
  value={text}
  onChangeText={setText}
  onSubmitEditing={handleSubmit}
  placeholder="Enter text"
/>

Two-way binding (from Lab 2)

Lab 2 — wk1_lab2, slide 35-36

By setting value={enteredText} on the input, you create two-way binding. The UI sends data to state via onChangeText, and state sends data back to the UI via value. This lets you programmatically reset the input (e.g. setEnteredText('') to clear after submit). 
const [enteredText, setEnteredText] = useState('');

<TextInput
  value={enteredText}
  onChangeText={(text) => setEnteredText(text)}
  placeholder="Type here"
/>

// Clear after save:
setEnteredText('');

6. useState — State Management

Lab 2 — wk1_lab2, slide 32-33

React ignores changes to regular JavaScript variables. To make data changes visible on screen, you must use State, which explicitly instructs React to re-render the component. useState returns an array: the current value and a setter function. 
import { useState } from 'react';

// Declare state
const [count, setCount] = useState(0);         // number
const [text, setText] = useState('');          // string
const [items, setItems] = useState([]);        // array

// Update (triggers re-render)
setCount(5);
setText('hello');

// Functional update (guaranteed correct previous value)
setCount(prev => prev + 1);

Immutable array updates

Lab 2 — wk1_lab2, slide 32

Never use .push(). It mutates the array but doesn't trigger a re-render. Use .concat() or spread [...prev, item] to create a new array. 
// ADD to array
setItems(prev => [...prev, newItem]);
// or
setItems(prev => prev.concat(newItem));

// REMOVE from array
setItems(prev => prev.filter(i => i.id !== id));

// UPDATE one item in array
setItems(prev => prev.map(i => i.id === id ? { ...i, done: true } : i));

7. Props — Passing Data Between Components

Lab 2 — wk1_lab2, slides 19-20

Props are custom attributes that pass data parent → child. Every functional component receives a single props object. Use curly braces {} to pass JS values instead of strings. 
// Parent passes data:
<GoalList goals={courseGoals} />

// Child receives via props:
const GoalList = (props) => {
  return props.goals.map(goal => <Text key={goal.id}>{goal.text}</Text>);
};

// Or destructure:
const GoalList = ({ goals }) => { ... };

Parent-child communication (lifting state up)

Lab 2 — wk1_lab2, slides 29-30

To send data upwards (child → parent), the parent passes a callback function as a prop. Convention: name it onSomething. The child executes it with data. 
// Parent:
const addGoalHandler = (newGoal) => {
  setCourseGoals(prev => prev.concat(newGoal));
};
<NewGoal onAddGoal={addGoalHandler} />

// Child:
props.onAddGoal(newGoal);  // triggers parent's function

8. Rendering Lists — .map() and FlatList

{items.map(item => (
  <Text key={item.id}>{item.text}</Text>
))}
Key prop is required. React needs a unique key for every list item to efficiently update the DOM. Usually use the item's id.

FlatList — efficient list for large data

<FlatList
  data={items}
  keyExtractor={(item) => item.id}
  renderItem={({ item }) => (   // NOTE: destructure { item }
    <Text>{item.title}</Text>
  )}
  ListEmptyComponent={<Text>No items</Text>}
  contentContainerStyle={{ padding: 16 }}
/>

13. Fetch — API Calls

Lab 7 — wk4_lab7, slides 13-16

const fetchData = async () => {
  try {
    const response = await fetch(url, options);  // 1st await: gets headers
    const result = await response.json();       // 2nd await: reads body
    setData(result);
  } catch (error) {
    Alert.alert('Error', error.message);
  }
};
Why two awaits? (From slide 14) This is a performance feature. It lets you check response.status (404, 500) before wasting time downloading the body.

Request configuration object

Lab 7 — slide 13

const options = {
  method: 'GET',       // or 'POST', 'PUT', 'DELETE'
  headers: {
    'Content-Type': 'application/json',
  },
  // body only for POST/PUT:
  body: JSON.stringify({ name: 'Test' }),
};

try...catch

Lab 7 — slide 15

From slides: High chance of failure when interacting with the outside world. Without a try block, an error will crash the app. The catch block prevents this — the error details are caught and handled instead.

Extracting data from response

Lab 7 — slide 16

// Text response:
const result = await response.text();

// JSON response:
const result = await response.json();

// Access a field (optional chaining from slides):
const quoteText = result[0]?.text || 'No quote found';
// ?. checks if result[0] is null/undefined before accessing .text

14. Component Rules

Lab 2 — wk1_lab2, slide 11 | Lab 3 — wk2_lab3, slides 12-16

1. Function name must start uppercase: HomeScreen not homeScreen
2. Must return renderable JSX
3. Must have one root element — wrap in <View> or <>...</>
4. Must have export default ComponentName at bottom
5. Every RN component must be imported from 'react-native'
6. <Button> is self-closing with title prop — not <Button>text</Button>
7. All text must be inside <Text> — raw text in <View> crashes the app
